
## Fly.io Migration & WebSocket Fix (Late Night Session)

### Completed
- Deployed Stellabot to Fly.io (`stellabot-app.fly.dev`)
- Fixed WebSocket 400 error - root cause was multiple WebSocketServer conflict
- Solution: Use `noServer: true` mode for machineWebSocket with manual upgrade handling
- Committed and pushed fix to GitHub (commit: 16b8c7f)

### Key Technical Finding
Multiple `WebSocketServer` instances on same `httpServer` can conflict. When one server's `verifyClient` returns false for a path it doesn't handle, it sends 400 before the correct handler can process. Fix: Use `noServer: true` and manually route upgrades.

## Stellabot → Clawdbot Connection Fix (18:00-18:12 PST)

**The final piece of the puzzle.** Web chat in Stellabot needed to reach Clawdbot on the Mac Mini.

### The Problem
Stellabot (on Fly.io) couldn't connect to Clawdbot (on Mac Mini). Logs showed:
```
❌ Clawdbot status fetch error: SyntaxError: Unexpected token '<', "<!doctype "... is not valid JSON
```

### Root Causes (Multiple!)

**1. Cloudflare Tunnel wasn't the answer**
- Originally tried `console.stellabot.app` via CF tunnel
- Tunnel worked for browser access but not for server-to-server from Fly

**2. Clawdbot wasn't listening on Tailscale interface**
- Default bind was `lan` (192.168.x.x) 
- Fly machines are on Tailscale mesh, need to reach via Tailscale IP
- **Fix:** Changed `gateway.bind` to `"tailnet"` in Clawdbot config

**3. Wrong API endpoint**
- Stellabot code was hitting `/api/status` which doesn't exist on Clawdbot!
- Clawdbot serves its dashboard SPA for all unmatched routes → returns HTML
- **Fix:** Changed health check to POST to `/v1/chat/completions` with invalid request
  - Returns 400 with JSON error = proves it's alive and reachable

### Solution Steps

1. **Clawdbot config change:**
```json
{
  "gateway": {
    "bind": "tailnet"  // Was "lan"
  }
}
```

2. **Fly secret update:**
```bash
fly secrets set CLAWDBOT_URL="http://100.74.241.116:18789" --app stellabot-app
```

3. **Stellabot code fix** (`server/routes/clawdbot-websocket.ts` and `debug.ts`):
```typescript
// Old (broken):
const response = await fetch(`${CLAWDBOT_URL}/api/status`, { ... });

// New (working):
const response = await fetch(`${CLAWDBOT_URL}/v1/chat/completions`, {
  method: 'POST',
  headers: { 'Authorization': `Bearer ${token}`, 'Content-Type': 'application/json' },
  body: JSON.stringify({ model: 'health-check', messages: [] }),
});
// 400 = reachable (expected), 200 = valid, both mean it's alive
const isAlive = response.status < 500;
```

### Network Path Now
```
Stellabot (Fly.io)
    ↓ Tailscale mesh
Mac Mini (100.74.241.116:18789)
    ↓ localhost
Clawdbot Gateway
```

### Key Learnings

1. **Clawdbot doesn't have `/api/status`** — Use `/v1/chat/completions` for health checks
2. **`gateway.bind: tailnet`** — Required for Tailscale peers to reach Clawdbot
3. **Tailscale over Fly.io** — Works great! Machines join the tailnet on boot
4. **Check what's actually running** — `curl -sI <url>` to see response codes before assuming what's available
