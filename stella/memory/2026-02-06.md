## Agent Security Architecture (Major Design Session)

### Context
John asked about creating a Brand Manager agent and ensuring proper guardrails. This led to a comprehensive architecture discussion about enterprise-grade agent permissions and multi-tenant isolation.

### Key Decisions

1. **Whitelist-Only Permissions**
   - Agents can ONLY do explicitly allowed actions
   - No blacklist approach (impossible to enumerate everything bad)
   - Organic growth: blocked attempts show what needs whitelisting

2. **agent_actions Table** (replaces complex multi-table approach)
   - One table: agent_id, action, constraints
   - Supports: specific actions, resource filters, value limits, time windows
   - Wildcard `*` for org admin agents
   - Protected actions list (sys.* never grantable)

3. **Multi-Tenant Isolation**
   - All enforcement at execution layer (immune to prompt injection)
   - org_id comes from authenticated session, never from agent input
   - Database: auto-inject org_id in all queries
   - Files: path jailed to `/data/orgs/{org_id}/`
   - Browser: per-org profiles `/data/browser-profiles/{org_id}/`
   - Credentials: lookup by org_id only

4. **Per-Org Browser Profiles**
   - Complete isolation between orgs (even on same machine)
   - Within org: agents share profile, permissions control what they can do
   - Cross-org: impossible to access other org's browser sessions
   - Solves OAuth credential leakage problem

## Hard vs Soft Agents Decision

**Key architecture decision from John:**

| Type | What it is | Examples |
|------|------------|----------|
| **Hard Agent** | Actual running process on infrastructure | Stella (Mac Mini), MoltWorker (CF Worker) |
| **Soft Agent** | DB config only — no dedicated process | Brand Manager, Org Admin, all org specialists |

**Rule: ALL non-sys-admin agents are soft agents.**

This means:
- No spinning up Clawdbot instances for every org agent
- Hard agents handle execution, soft agents are just config
- `machines` table only for hard agents
- Scales to thousands of soft agents with handful of hard agents
- Soft agents invoked/orchestrated by hard agents when needed

Updated `AGENT-SECURITY-ARCHITECTURE.md` with this distinction.

## Google Integration Model

**One integration account per org** (e.g., stella@killerapps.dev). Agents don't need their own Google accounts.

**Two permission layers:**
1. **Google side:** Resource must be shared with the integration account
2. **Stellabot side:** Agent must have action whitelisted

Both must pass. Clean separation — Google handles resource sharing, Stellabot handles agent permissions.

Example: Brand Manager accessing spreadsheet
1. Spreadsheet shared with stella@killerapps.dev (Google)
2. `sheets.read` + `sheets.write` whitelisted for Brand Manager (Stellabot)
3. Agent can access it

## Architecture Decision (14:48 PST)

Major architectural clarity achieved:

**The Model:**
- **App Sys Agent** = brain running in Stellabot cloud, direct Anthropic API
- **Soft Agents** = just prompts/personas, no runtimes
- **Clawdbot** = ONE tool (hardware access only), not the brain
- **Cloud tools** = Sheets, Slack, DB all called directly by Sys Agent

**Key insight from John:** "Clawdbot becomes 'a' tool with an API, not THE tool"

**Why this is better:**
- Stellabot works without any Clawdbot
- Clawdbot is optional for users wanting local machine access
- No single gateway bottleneck
- Clean separation: cloud stuff in cloud, hardware stuff via Clawdbot
- Scales independently

Spec written: `stellabot-replit/docs/ARCHITECTURE.md`
