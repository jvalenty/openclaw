## Evening Session: Architecture Decisions

### Reseller Architecture (Decision Checkpoint)

**Problem:** Agent OAuth failing for Stella because she had no org. Deeper question: how do platform sys admins relate to orgs?

**Decision:** Uniform Reseller Model
- Every org is first-class (no special "platform" tier)
- Machines belong to orgs (owner)
- Sys admins belong to machine's org
- Machines can be authorized to serve multiple orgs (multi-tenant)
- Any org can become a reseller (e2e is just the first)

**Key insight from John:** "Orgs can also be resellers exactly like e2e. Perhaps there should be zero diffs."

**Spec written:** `docs/specs/reseller-architecture.md`

**Solution:** Assign Stella to e2e org. Her secrets store there. She can still serve all orgs via machine authorizations.

## 9:55 PM - Session Jam Incident

**What happened:**
1. Hit Anthropic API usage limits around 6:11 PM PST
2. Auth profile went into cooldown
3. Heartbeats kept firing, each creating new failed sessions → 342 session pileup
4. John upped limit, I woke up
5. Started thrashing on bad DB queries (wrong column names, local psql instead of Neon)
6. 10-minute timeout mid-tool-call
7. Session history corrupted (tool_result without matching tool_use)
8. Every retry failed due to corrupted history
9. John had to clear all sessions manually

**Lessons learned:**
- **NEVER use bare `psql`** — always use Neon connection string: `psql "$NEW_DB"` where NEW_DB is set in .zshrc
- **Verify column names before SQL** — `org_id` not `organization_id`, check schema first
- **Session corruption happens on timeout** — if mid-tool-call when timeout hits, history breaks
- **Heartbeat pileup risk** — when auth is in cooldown, heartbeats still fire and create zombie sessions

**Prevention:**
- Before any DB query, verify I'm using correct connection method
- If hitting errors repeatedly, STOP and reassess instead of thrashing
- If something takes >5 min, check if approach is wrong
