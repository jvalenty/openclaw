# 2026-03-01

## Incident: Opus Token Burn & Secret Exposure
- John noticed Opus tokens burning despite the new secrets config missing the Anthropic key.
- **Root Cause Found:** `~/.openclaw/openclaw.json` `env.vars` had the old Anthropic key hardcoded. OpenClaw loads this config at startup (via `shellEnv.enabled: true`), overriding the clean LaunchAgent plist.
- **My Mistake:** While debugging, I accidentally dumped all four provider keys (ANTHROPIC, OPENAI, GOOGLE, BRAVE) into the session context via tool output. I immediately flagged this to John for rotation.
- **Immediate Fix:** John deleted the old vars and restarted the gateway.

## Architectural Decision: Secret Injection
- "Apply to Machine" was injecting provider keys into `openclaw.json env.vars`, which led to plaintext secrets on disk in our workspace.
- **Options debated:**
  1. Write to LaunchAgent plist (still plaintext on disk)
  2. Write to `~/.openclaw/.env` (better, native OpenClaw fallback, but still on disk)
  3. **Chosen Path:** OpenClaw SecretRefs + `source:"exec"`. A local resolver script calls Stellabot's `/api/machine-secrets/resolve` using the existing machine token. Zero disk secrets.
