## 🔄 Architecture Flip - Late Night Session (22:22-23:36)

**John's directive:** "Web/DB has to be OURS and machines need to be handlers"

### The Problem
- Clawdbot was source of truth for chat history
- Web chat and Telegram were separate sessions
- History lived in 3rd party system (Clawdbot), not our DB

### The Solution - Flipped Architecture

**Before (wrong):**
```
Telegram → Clawdbot (owns channel, stores history) → Response
Web Chat → Clawdbot (separate session) → Response
```

**After (right):**
```
Telegram → STELLABOT (owns channel, saves to DB) → Clawdbot (stateless) → STELLABOT (saves response) → Telegram
Web Chat → STELLABOT (saves to DB) → Clawdbot (stateless) → STELLABOT (saves response) → Web
```

### Key Changes

1. **Web Chat (`clawdbot-chat.ts`)**
   - Saves user messages to DB BEFORE calling Clawdbot
   - Loads full conversation history FROM DB
   - Passes history to Clawdbot (stateless call)
   - Saves assistant response to DB

2. **Telegram Handler (`channels.ts`)**
   - New webhook endpoint: `POST /api/channels/telegram/webhook/:token`
   - Setup endpoint: `POST /api/channels/telegram/setup/:id`
   - Send endpoint: `POST /api/channels/telegram/send`
   - Test endpoint: `GET /api/channels/telegram/test/:id`
   - Same pattern: Save to DB → Load history → Call Clawdbot → Save response → Send to Telegram

3. **Documentation**
   - `docs/CHAT-SYNC.md` - Full architecture docs
