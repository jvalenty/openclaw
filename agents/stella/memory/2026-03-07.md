## Dan/Stellabot Screenshot Upload Bug
**Symptom:** Dan (soft agent in Earnware org) or John is uploading screenshots via the Stellabot chat UI, but they just show up as `[Screenshot]` with no actual image attached. Bella reproduced this: attaching an image in Earnware org results in Dan saying "I don't see an attached image" and the image is not relayed properly.

**Investigation Context:**
- In the frontend `GlobalChatSidebar.tsx`, the `requestPayload.images` array is populated from `attachments`.
- The `POST /api/soft-agent/:agentId/chat/sync` endpoint processes this `images` array via `captureUserFile` (which inserts into the `files` table and R2).
- The `[Screenshot]` text usually comes from `captureAgentFile` or the initial message insertion.
- The Slack relay (`server/routes/channels.ts`) attempts to fetch the image from `/api/agent-files/:fileId` using `x-internal-channel-call` headers, then uploads it to Slack.
- The bug likely stems from the recent Capability Isolation / `GAP-1` security middleware fixes (which strictly enforces `X-Org-Id`) blocking either the initial image upload or the subsequent internal fetch by the Slack relay, OR a frontend regression in how `images` are passed to the `/chat/sync` endpoint after the `ChatInput` refactor (commits `0ee79db` / `b0be629`).

**Pending Next Steps:**
1. Check the frontend network payload for `POST /api/soft-agent/:agentId/chat/sync` to ensure `images` are actually being sent.
2. Verify if the backend `captureUserFile` is throwing an error or failing to upload to R2 because of org ID scoping.
3. Check the internal fetch in `channels.ts` to see if it's getting a 401/403 due to the new security middleware missing the `X-Org-Id` header.
4. John specifically requested I (Stella) step in to help fix this without guessing, relying on actual logs and network traces.
