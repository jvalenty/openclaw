## Agent Architecture Cleanup

### Problem
Clawdbot had 4 agents registered (Stella, Paul, Jarvis, Brand Manager) when only Stella should be a "hard agent". This caused architectural confusion about how soft agents should work.

### Correct Architecture

**Hard Agents (run on Clawdbot):**
- Have physical machine access (files, browser, exec, etc.)
- Registered in Clawdbot's `agents.list` config
- Example: Stella on Mac Mini

**Soft Agents (run in Stellabot):**
- Context stored in Stellabot database (`agents.modelConfig.systemPrompt`)
- Execute via Claude API directly from Stellabot
- Have cloud tools (Sheets, Calendar, Gmail, Slack)
- Route to hard agents when hardware access needed
- Examples: Paul, Jarvis, Marcus, Asaph, Nova, Roxanne, Brand Manager

### Routing Logic (agent-chat.ts)
```javascript
const isSoftAgent = agent && 
  agent.agentType !== 'sys_admin' &&  // sys_admin = hard agent
  !agent.machineId &&                  // no machine = soft
  !agent.endpoint &&                   // no custom endpoint
  agentId !== 'stella' &&              // special cases
  !agentId.startsWith('machine:');     // machine proxies
```

## Conversational Agent Architecture - SHIPPED

Built and deployed the dual-loop conversational architecture John requested:

### What It Solves
- **Message serialization** - rapid-fire messages were losing context
- **Blocking on work** - conversations stalled during long tasks
- **No interrupts** - user couldn't change direction mid-task

### Architecture
```
User → Batch Window (1.5s) → Conversational Agent (Haiku - fast)
                                    ↓
                    Quick Response OR Dispatch to Work Queue
                                    ↓
                            Work Agents (async)
                            
Meanwhile: New messages can interrupt running work
```

### Files Created
- `shared/models/conversation.ts` - Drizzle schema for 3 tables
- `server/services/conversation-loop.ts` - Core orchestration
- `server/routes/conversation.ts` - API endpoints + SSE
- `migrations/0012_conversation_loop.sql` - Database migration
- `docs/architecture/CONVERSATIONAL_AGENT_DESIGN.md` - Full spec
- `docs/architecture/IMPLEMENTATION_STATUS.md` - Rollout plan

### Database Tables
- `message_batches` - Groups rapid messages
- `work_queue` - Async task dispatch + tracking
- `conversation_state` - Rolling context window

### API Endpoints
- `POST /api/conversation/sessions` - Get/create session
- `POST /api/conversation/:sessionId/message` - Batched message intake
- `GET /api/conversation/:sessionId/state` - Current state
- `GET /api/conversation/:sessionId/work-stream` - SSE for real-time
- `POST /api/conversation/:sessionId/work/:workId/interrupt` - Cancel work
