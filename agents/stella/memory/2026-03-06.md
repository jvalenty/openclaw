## Bug 2: Attachments empty at send time — ROOT CAUSE STILL UNKNOWN

### What we know:
- `[CHATINPUT-DEBUG] handleFileSelect called, files: 1 externalFileSelect: true` → fires TWICE
- `externalFileSelect` is truthy and IS called
- `[ATTACH-DEBUG-ENTRY]` (first line of GlobalChatSidebar's `handleFileSelect`) NEVER appears
- `[ATTACH-DEBUG]` (after processFile) NEVER appears
- `[SEND-DEBUG] attachments at send time (via ref): 0 []` — ref-based approach still 0

### The paradox:
`externalFileSelect: true` → it IS called → BUT `[ATTACH-DEBUG-ENTRY]` at the very first line of `handleFileSelect` never appears. This means either:
1. `externalFileSelect` is not actually pointing to GlobalChatSidebar's `handleFileSelect`
2. The function IS called but logs are suppressed somehow (different component instance? stale bundle?)

### Latest deploy (51ebba7) adds:
- `[ATTACH-DEBUG-ENTRY]` at very first line of `handleFileSelect` (before any async)
- `[ATTACH-DEBUG] processFile ERROR:` in catch block
- `[CHATINPUT-DEBUG]` now wrapped in try/catch and logs file type

### Next step when Bella tests:
Look for `[ATTACH-DEBUG-ENTRY]`. If it doesn't appear → `externalFileSelect` is NOT calling GlobalChatSidebar's `handleFileSelect`. Most likely cause: there are TWO instances of GlobalChatSidebar mounted (or the component remounts), and the `onFileSelect` prop captured by ChatInput belongs to an old/different instance.
