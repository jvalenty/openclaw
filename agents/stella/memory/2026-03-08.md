## browser_login fix chain (3 bugs, 3 deploys)

### Bug 1: Credential format mismatch (40a4ffb)
**Root cause:** UI sends `{ service, secretType, data: { username, password } }` to
`/api/agent-secrets/:agentId`. Handler spreads `...secretData` = `{ data: {...} }` and
stores the wrapper. `getSecret` returns `{ data: { username, password } }`. Then
`getBestAuthMethod` checks `credentials.password` — undefined because it's nested at
`credentials.data.password`. Login fails with "no credentials stored."

**Fix:**
- `agent-secrets.ts` POST: unwrap `secretData.data ?? secretData` before `setSecret()`
- `browser-auth.ts` `getBestAuthMethod`: `const creds = (auth.credentials as any)?.data ?? auth.credentials`
  so both old and new format work (backwards compatible)

**DB evidence:** `app.earnware.ai:credentials` preview was `{"dat...!!"}`
not `{"use..."}` — data wrapper was there, matching the analysis.

### Bug 2: Machine Service not configured (047c73d)
**Root cause:** `performLogin()` in `browser-auth-api.ts` read from
`process.env.MACHINE_SERVICE_URL` / `MACHINE_SERVICE_TOKEN` — not set on Fly.
Screenshots/nav worked because they use `resolveMachineService()` which queries
the DB `machines` + `machine_authorizations` tables.

**Fix:**
- `executeBrowserLogin` in `soft-agent-chat.ts` now calls
  `resolveMachineService(undefined, ctx.orgId, { sessionId: ctx.sessionId })`
  before calling `performLogin`
- `performLogin` signature extended to accept `machineUrl?, machineToken?, cfAccessHeaders?`
- `executeLoginSteps` also extended to accept and forward `cfAccessHeaders`
- Env var fallback retained for backwards compatibility

- Earnware (`22377b4f`) authorized in `machine_authorizations` with `enabled=true`
