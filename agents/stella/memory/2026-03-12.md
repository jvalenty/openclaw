
## Group Threads Visibility Bug (2026-03-12)

**Symptom:** Org Admins (like Sophia) could not see Group Threads they were explicitly added to, whereas Sys Admins (like Bryan) could.

**Root Cause 1 (Backend - Fixed by Bella):** 
The `/api/group-threads` endpoint had an implicit fallback: `const selectedOrgId = req.query.orgId || user.orgId;`. 
Because Sophia didn't send an explicit `orgId` (since she doesn't have an Org Switcher dropdown by default), it fell back to her `user.orgId`. The query then filtered out any threads where `groupThreads.orgId !== user.orgId`. The "Brand Managers" thread was created by Bryan under his org, so it was silently filtered out for Sophia despite her being in the `groupThreadMembers` table.
*Fix:* Bella updated the endpoint to only apply the `orgId` filter if explicitly provided via query param (`requestedOrgId`), making `groupThreadMembers.userId` the primary access control.

**Root Cause 2 (Frontend - Fixed by Stella):** 
After deploying the backend fix, Sophia STILL couldn't see the thread. The `useSelectedOrg` hook reads from `localStorage('stellabot_selected_org')`. If Sophia previously had a stale org ID (like the System org from when John toggled her to Sys Admin), the client was sending that stale ID in `?orgId=`. Because she wasn't an admin anymore, the OrgSwitcher didn't render to let her change it, and the `useEffect` only auto-selected if the stored ID was entirely missing (falsy).
*Fix:* I updated `useCurrentOrg` in `OrgSwitcher.tsx` to validate that the `selectedOrgId` actually exists in her `accessibleOrganizations` list. If not, it forcefully resets to her first valid org. 

**Deployment:** Both fixes are merged to `main` and deployed to `stellabot-app.fly.dev`. We are waiting for John to confirm Sophia can now see the "Brand Managers" thread.

*Note:* During debugging, I used `fly logs` and `node -e` scripts against `/app/dist/index.cjs` to confirm the actual minified bundle matched the expected source code.
